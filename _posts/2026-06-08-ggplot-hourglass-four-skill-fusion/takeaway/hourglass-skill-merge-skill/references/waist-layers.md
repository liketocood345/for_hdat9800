# Waist layers — what each segment decides

| Layer | When | Decides |
|-------|------|---------|
| **Inverse tree** | Entry | Task class (`mode`), stack, overlay hints; usually **not** final deliverable subtype |
| **Waist Z0 `mode`** | After inverse | Positive-tree root (explore / manuscript / review / gallery…) |
| **T0 / Z1 `data`** | After profile | Feasible deliverable family; feeds Γ |
| **BIND / Z2 `chart`** | After Γ + δ | Core deliverable choice + leaf id |
| **Z3 `delivery`** | BIND or render | Export format, bundle-only, dimensions |
| **Belt `overlay[]`** | Cross-cutting | Style/compliance rules orthogonal to tree |
| **Positive tree** | After waist commit | Open which upstream file + section |
| **Leaf** | Terminal | Executable template / CLI from **linked** repo |

**Important:** chart (or domain-critical field) may be **most important semantically** but **commits late** — after mode and data feasibility narrow options.

**P0 shortcut:** explicit user utterance can pre-fill a waist field at inverse tree (e.g. `chart: heatmap`).
