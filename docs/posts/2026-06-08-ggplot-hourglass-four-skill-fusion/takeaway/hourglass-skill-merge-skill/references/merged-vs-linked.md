# Merged vs Linked

## Linked (~100% of deliverable body)

- Upstream clones stay intact: `SKILL.md`, scripts, templates, CLI
- Fusion **never** rewrites template code into the fused skill
- At runtime: open **one** leaf → read upstream `§` section

## Merged (~0% upstream body copy)

New artifacts only:

- Thin fused `SKILL.md` (orchestration + Do-not rules)
- `hourglass-waist-registry.yaml` (entries, gamma, mutex, leaves)
- `references/*.md` (shards for progressive disclosure)
- Optional diagrams / viewer HTML

## Mount vs read

| | Mount | Read each turn |
|---|-------|----------------|
| Meaning | All upstream skills registered in `skills:` | Load full upstream `SKILL.md` bodies |
| Hourglass | Yes — topology knows four repos | **No** — one leaf after BIND |

## Is fusion just an index?

**Partly:** leaf table and entry aliases are index-like.

**Also new law:** mode taxonomy, BIND+Γ, belt, mutex, conflict_matrix, phased waist commit, STOP edges — none of which exist in any single upstream skill.

**Formula:** fused skill = **single-entry router + commit protocol + conflict matrix + maintainer map**; index is one layer.
